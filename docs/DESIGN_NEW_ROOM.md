# DESIGN: 新建房间流程 — 交互优化

版本: v1.0
日期: 2026-07-22
状态: 📐 design-ready

---

## 现状 & 问题

```
现在: 点 FAB ＋ → Sheet 弹出 → 填名字 → 选议会/单聊 → 创建 → 丢进房间

问题:
  1. "议会"/"单聊"是工程师术语。用户不知道区别
  2. 议会描述里写了硬编码的 claude/gpt-5/gemini, 和用户实际配的模型无关
  3. 创建后 Hermes 的欢迎语是通用的, 没有围绕用户的项目
  4. 默认名"新项目 3"毫无信息量
  5. 没有引导——用户不知道建完房间该做什么
```

---

## 1. 设计目标

三层递进：**建前有理由 → 建时低摩擦 → 建后有方向。**

- 建前：用户知道"我为什么要创建这个房间"
- 建时：一张 Sheet，两步，无技术术语
- 建后：Hermes 围绕用户的项目给出具体的第一句话

---

## 2. Sheet 设计

### 第一步：基本信息

```
┌─────────────────────────────────────┐
│ 新建项目房间                   [✕]  │
│                                     │
│ 项目名                              │
│ ┌───────────────────────────────┐   │
│ │ 产品 V2.0 方案讨论             │   │
│ └───────────────────────────────┘   │
│                                     │
│ 简单描述 (选填, 帮助 AI 理解项目)     │
│ ┌───────────────────────────────┐   │
│ │ 讨论新版本功能优先级, 确定      │   │
│ │ P0/P1/P2, 输出 PRD 初稿        │   │
│ └───────────────────────────────┘   │
│                                     │
│ 下一步: 选择协作方式 →              │  ← 这不是按钮, 是提示
│                                     │
│          [ 取消 ]   [ 下一步 ]       │
└─────────────────────────────────────┘
```

描述不是必填——但给用户机会说清"这个房间要干什么"。如果填了，这个描述自动变成房间的第一条系统消息 + Hermes 的第一句话会围绕它。

### 第二步：选择协作方式

```
┌─────────────────────────────────────┐
│ ← 返回   选择协作方式               │
│                                     │
│ ┌─────────────────────────────────┐ │
│ │ ● 和 MOV 一对一                 │ │  ← 默认选中
│ │   快速问答、设备指令、简单记录    │ │
│ │   ～～～～～～～～～～～～～～～  │ │
│ │   房间里只有你和 MOV              │ │
│ │   适合: 写笔记、问问题、查资料    │ │
│ └─────────────────────────────────┘ │
│                                     │
│ ┌─────────────────────────────────┐ │
│ │ ○ 拉 AI 团队一起讨论             │ │
│ │   多角色给不同视角, 投票决策      │ │
│ │   ～～～～～～～～～～～～～～～  │ │
│ │   从你配置的 AI 中选参与者         │ │  ← 关联到 ≡ 设置里的 AI 团队
│ │   适合: 方案评审、头脑风暴、技术选型│ │
│ └─────────────────────────────────┘ │
│                                     │
│          [ 上一步 ]   [ 创建 ]       │
└─────────────────────────────────────┘
```

**关键变化**：
- "议会"→"拉 AI 团队一起讨论"
- "单聊"→"和 MOV 一对一"
- 议会描述不再写死 claude/gpt-5/gemini，改为"从你配置的 AI 中选参与者"
- 一对一默认选中——降低决策负担
- 两步结构：先填信息 → 再选方式。用户可以在第二步"上一步"回去改。

## 3. 创建后的第一句话

### 如果用户填了描述

```
用户在一步填了: "确定 P0/P1/P2, 输出 PRD 初稿"
选了: 拉 AI 团队

→ 系统消息: "项目「产品 V2.0 方案讨论」已创建。
           目标: 确定 P0/P1/P2, 输出 PRD 初稿"

→ Hermes: "好的。我建议先从功能清单开始——
          要不要把你脑子里的功能先列出来，我帮你分类？"
```

### 如果用户没填描述

```
用户只填了名字: "产品 V2.0", 没填描述
选了: 和 MOV 一对一

→ Hermes: "「产品 V2.0」已就绪。
          直接发指令、提问、或者让我帮你写东西都行。"
```

**第一句话的策略**：
- 有描述 → 围绕描述给出下一步建议。是一个具体的动作邀请。
- 没描述 → 简洁告知就绪。不废话。
- 议会模式 → 额外说明"AI 团队已就绪，请提出议题"。

## 4. 需要改的文件

| `hermes-shell.html` | Sheet 改成两步结构；两个协作卡片的新文字 | 中 |

- Java 层：零改动
- 创建房间的底层逻辑：不改
- 房间数据模型：不改

## 5. 默认名策略

```
用户什么都没填 → "新项目" (不带数字)
用户填了描述 → 取描述前 15 字作为默认名 (如果没填名字)

输入框 placeholder: "项目名 · 不填就用描述的前几个字"
```

"新项目 3" → "新项目"。数字没有信息量，不如不加。
