版本: v4.0

## 0. 一句话定义

> Android 手机上，拉多个 AI 模型进房间一起干活。

---

## 0.1 设备

**手机。** 竖屏。单手操作。全文统一，不再出现"平板"。

## 0.2 看板为什么存在

看板不是娱乐中心。它是**轻应用桌面**——房间产出的落地终端。

```
房间：AI 写了一个健身 APP → 部署到看板 → 用户实际使用
看板：笔记里写的方案 → 发送到房间 → 变成协作素材
```

看板留在 MOV 里的理由：同一台设备上，用户既有协作需求也有个人工具需求。拆成两个 App 的代价（切换、数据不通）大于留在同一个 App 里的认知负担。

边界规则：
- 看板应用数据是私有的，不进入房间
- 用户可以主动"发送到房间"（笔记→聊天消息），但不会自动
- 房间里的 AI 不能访问看板数据

## 2. 会话 — 项目房间

### 房间 = GitHub Repo 的样子

```
房间: 产品 V2.0
├─ [讨论]                        ← 聊天。人说话，AI 说话。MOV 主持。
│   └─ 消息长按删除
│   └─ Council 投票卡
│   └─ 文件卡片 (AI 产出直接嵌入讨论流)
│
├─ [文件]                        ← 存储系统 v3.0
│   └─ [产出] AI 和人共同编辑的文件 (版本树)
│   └─ [资料] 只读参考素材 (网格视图)
│   └─ [归档] Cron 自动产出的报告 (按来源分组)
│   └─ [模板] 跨房间复用的格式 (使用=复制)
│
└─ /sdcard/mov/rooms/<id>/   ← 真实磁盘目录
    ├─ files/work/            产出 (当前版本)
    ├─ files/work-snapshots/  产出历史版本
    ├─ files/inbox/           资料
    ├─ files/archive/         归档 (按来源分目录)
    ├─ files/.meta/index.json 元数据
    └─ .mov/config.json       房间配置
```

### 房间类型

| 类型 | 参与者 | 场景 |
|------|--------|------|
| 单聊 | MOV (1 个 AI) | 快捷指令、简单问答 |
| 议会 | MOV + 多个 AI 模型 | 产品讨论、方案评审 |

### 房间操作

| 操作 | 入口 | 交互 |
|------|------|------|
| 创建 | FAB ＋ → Sheet | 填名称 + 选模式 (议会/单聊) |
| 重命名 | ⋮ / 长按房间卡片 → Sheet | Sheet 内输入框 |
| 归档 | ⋮ / 长按房间卡片 → Sheet | 直接执行 + toast |
| 清空聊天 | ⋮ / 长按房间卡片 → Sheet | Sheet 确认态 |
| 删除 | ⋮ / 长按房间卡片 → Sheet | Sheet 确认态 (红色) |

## 3. 看板 — 轻应用面板 (v2.0 悬浮切换)

### 交互模型

```
全屏内容 (iframe)
  → 底部触发条 (3秒自动隐藏)
  → 点击触发条 → 应用选择面板 (3列网格)
  → 选应用 → 面板关闭 → 内容切换
  → 触碰底部 80px 区域 → 触发条唤醒
```

### 自带应用

| 应用 | 文件 | 功能 |
|------|------|------|
| 音乐 | `board-apps/music.html` | 播放器 UI + /sdcard/music/ |
| 阅读 | `board-apps/reader.html` | 暖纸色文本阅读器 |
| 健身 | `board-apps/fitness.html` | 训练计时器 (深色) |
| 笔记 | `board-apps/notes.html` | 等宽字体编辑器 + localStorage |

### 用户添加

面板底部 ＋ → Sheet (名称+URL) → 添加到网格。长按可删除。

## 7. 交互规范

| 模式 | 适用场景 | 行为 |
|------|---------|------|
| 点击 | 进入/打开 | 直接执行 |
| 长按 (500ms, 移动>10px 取消) | 删除/操作 | 高亮 → 底部黑条浮出 → 点击执行 |
| Sheet | 创建/编辑/确认 | 遮罩 + 底部面板滑入, ✕ 关闭 |
| Toast | 操作反馈 | 短暂弹出, 自动消失 |
| Overlay | 预览/版本历史 | 全屏遮罩 + 内容面板, ✕/遮罩关闭 |

**原则**: 每操作必有反馈。能创建就能删除。触屏优先, 拒绝 prompt()/confirm()。

# MOV — 运行状态 & 使用场景

版本: v1.0
日期: 2026-07-22

---

## 最终运行状态

MOV 常驻在你的 Android 手机上。它不是"打开用一下然后关掉"的工具——它是**一直在运行的工作台。** Cron 定时任务在你睡觉时执行；AI 模型随时待命；设备互联后，PC 上的 GPU 训练任务状态实时显示在你的手机上。

### 打开 MOV，你看到三个 tab

**会话** — 你的项目房间列表。每个房间是一个项目。房间里有人（你、同事）和 AI（你配置的多个大模型）。讨论区里人和 AI 的对话交织在一起——AI 不只是回答问题，它们有角色、有观点。文件区里是 AI 生成的代码、同事上传的 PDF、Cron 自动产出的每日报告。点进一个文件，你能看到它是什么时候、在哪次讨论中、由谁创建的。

**看板** — 你的个人空间。底部一条半透明胶囊条显示当前应用，3 秒不动自动隐去。点一下弹出应用面板——音乐、阅读、健身、笔记，还有你自己加的。内容全屏，没有 chrome。这是你一个人的地方，和协作无关。

**运行** — 你的设备集群面板。MOV 在运行、AI 模型在线上、定时任务在跑。未来 PC 接进来，GPU 训练进度条实时跳动。通道状态折叠成一行摘要——全部正常时你扫一眼就走了。有问题才展开。

---

## 四个核心使用场景

### 场景一：让 AI 团队帮你做决策

你有一个产品方案要评审。打开 MOV，创建一个房间叫"产品 V2.0"，写了一句描述："确定 P0/P1/P2 优先级，输出 PRD 初稿"。选择了"拉 AI 团队"，勾选了 DeepSeek（通用视角）和 Claude（技术视角）。

你发了一条消息："这是当前的功能清单，帮我评估优先级。"

DeepSeek 和 Claude 并行回复——一个从产品角度说"注册登录是 P0"，一个从技术角度说"支付模块依赖第三方接口，风险高"。Hermes 汇总："共识：P0 包含注册登录和商品浏览。分歧：支付模块的优先级。"

讨论了两轮，你说"就按这个来，帮我写 PRD"。Hermes 提取步骤，逐条执行——DeepSeek 写初稿，Claude review，最终产出 `PRD-v2.md` 存入房间文件区。

**这不是聊天。这是决策 + 执行。**

### 场景二：把 AI 当同事一起写代码

你在"健身 APP"房间里。之前已经和 AI 讨论过架构。现在你说"帮我写登录页面，支持手机号+验证码"。

DeepSeek 生成 `Login.tsx`，存入文件区。你看了觉得可以，但需要加表单校验。你说"加个手机号格式校验"。Claude 在 DeepSeek 的版本上修改，生成 v2。你手动改了三行标题。DeepSeek 又出了一版优化性能。

现在 `Login.tsx` 有 4 个版本，版本树里能看到：v1 DeepSeek 生成 → v2 Claude 修改 → v3 你手动改 → v4 DeepSeek 优化。你可以随时回到 v2 看 Claude 当时改了什么，也可以对比 v3 和 v4 的差异。

**文件不是终点——是协作的产物。每个版本都有来源、有上下文。**

### 场景三：一个人时的个人空间

你在看板 tab。点开音乐，播放本地 MP3。切到笔记，写下今天的想法——停写 2 秒自动保存。你想到一个方案，觉得可以拿到项目房间讨论——长按笔记，点"发送到房间"，内容变成一条聊天消息。

健身应用里记录了你今天跑了 5 公里。月底回看月历，15 个 🏃 图标散布在日历上。

**这些和协作无关。这是你自己的空间。私有的，不和 AI 共享。**

### 场景四：让 MOV 自己干活

你设了三个 Cron：
- 每天早上 8:30 汇总未读邮件 → 产出 `7月22日-邮件汇总.md`，存到房间归档区
- 每周一检查腾讯云账单 → 产出消费报告
- 每小时检查三个网站是否在线 → 异常时发通知

你不需要打开 MOV 来执行这些——它们自动跑。你只需要在归档里翻看历史产出，或者在运行 tab 看到"Cron 全部正常"的绿点。

以后 PC 接入后，你可以在手机上看到 GPU 工作站上的 LoRA 训练进度条——85%，预计 12 分钟。

**MOV 不是打开才工作。它一直在运行。**
# DESIGN: 产品定位

版本: v1.0
日期: 2026-07-22

---

## 定死：手机。

之前 CLAUDE.md 写"平板"，MOV_VISION.md 写"手机"——来回摇摆。现在定死。

- 设备：Android 手机，竖屏为主
- 交互：单手操作，拇指热区优先
- 信息密度：一屏一个核心任务，不堆叠

---

## 定死：协作 + 个人双场景。

MOV 不是纯协作工具，也不是纯个人助手。它服务于同一台设备上的两种需求：

| 场景 | 在哪 | 核心任务 |
|------|------|---------|
| 协作 | 会话 tab | 拉 AI 团队进房间讨论、决策、产出文件 |
| 个人 | 看板 tab | 使用轻应用（笔记、健身、音乐）做自己的事 |

**看板不是娱乐中心。** 健身应用是训练计时器，不是看视频。音乐应用是播放器，不是刷短视频。这些是工具，不是消遣。

---

## 定死：看板保留，但改定义。

从"个人空间"改为"轻应用桌面"。和协作的关系：

```
房间里 AI 产出了一个健身 APP → 部署到看板 → 用户实际使用
看板上发现笔记有价值 → 发送到房间 → 变成协作素材
```

**看板是房间产出的落地终端。** 不是独立的"个人娱乐区"。
