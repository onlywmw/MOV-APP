# MOV 存储系统 — 用户视角设计

版本: v3.0
日期: 2026-07-22
状态: 📐 design-ready

---

## 为什么存储设计是 MOV 的基石

用户每天打开 MOV，做的每一件事都依赖存储：

- 进房间 → 加载聊天记录
- 点文件 tab → 看到项目资产
- 搜"上次 AI 写的登录页面" → 跨时间检索
- 看板上写笔记 → 保存个人内容
- Cron 跑了三个月 → 翻看历史产出

**存储设计错了 → 用户找不到自己的东西 → 他不会继续用。** 这不是技术选型问题，是产品能否存活的问题。

---

## 一、五种存储，五种体验

MOV 用户产生五类完全不同的数据。它们不是"文件"的同义词——它们是五种**使用场景**。

| | 产出 | 资料 | 归档 | 模板 | 个人 |
|------|------|------|------|------|------|
| **一句话** | AI 和我一起写的代码和文档 | 甲方发的、同事传的参考素材 | Cron 每天自动生成的报告 | 做过一次、下次还用的格式 | 自己写的笔记、听的歌、练的健身 |
| **范围** | 房间内 | 房间内 | 房间内 | 跨房间 | 设备级，私有 |
| **来源** | AI 生成 + 人修改 | 人从设备上传 | Cron 定时产出 | 人创建 | 人 |
| **能改吗** | AI 和人能改 | 不能 | 不能 | 创建者能改 | 能 |
| **有版本吗** | 版本树 | 无 | 每次新文件 | 版本 | 无 |
| **谁可以删** | 房间创建者 | 上传者 + 创建者 | 房间创建者 | 创建者 | 自己 |
| **会过期吗** | 不 | 不 | 保留策略 | 不 | 不 |
| **用户怎么找** | "上次谁改了那个页面" | "甲方发过一份支付相关的 PDF" | "7月15号的邮件汇总" | "我做的 PRD 模板" | "上周写的产品想法" |

## 二、产出 — AI 和我一起写的代码和文档

### 用户场景

产品经理打开"产品 V2.0"房间。过去三天，他和张三、Claude、DeepSeek 讨论了登录功能。AI 生成了好几版代码，他自己改过，张三给过意见。

他打开文件 tab，想找到两天前 Claude 改的那版 Login.tsx，看看 Claude 改了什么，决定要不要用那版替代当前的。

**他不是在"管理文件"——他是在追溯决策过程。**

### 默认视图：按时间

```
┌─ 产出 ────────────────────────────────────┐
│ [时间] [作者] [来源] [类型]    🔍 搜索…   │
│                                           │
│ ▼ 今天                                    │
│ ┌─── 📄 Login.tsx ──────────────────────┐ │
│ │    最新 v4 · DeepSeek · 2h 前          │ │
│ │    4 个版本 · 有 1 个并行修改待处理     │ │
│ │    来源: "写个登录页面" →              │ │
│ └───────────────────────────────────────┘ │
│ ┌─── 📄 utils.ts ───────────────────────┐ │
│ │    最新 v2 · Claude · 1h 前            │ │
│ └───────────────────────────────────────┘ │
│                                           │
│ ▼ 昨天                                    │
│ ┌─── 📄 PRD-v2.md ──────────────────────┐ │
│ │    最新 v5 · Council 投票通过 · 昨天   │ │
│ │    5 个版本 · 来源: 产品方案评审会议    │ │
│ └───────────────────────────────────────┘ │
│                                           │
│ 共 7 个文件 · 14 个版本 · 128KB           │
└───────────────────────────────────────────┘
```

### 点进一个文件：版本树

```
┌─ Login.tsx ─────────────────────────────────┐
│                                              │
│ 当前: v4 · DeepSeek · 2h 前                  │
│ ┌──────────────────────────────────────────┐ │
│ │ import { useState } from 'react';        │ │  ← 代码视图
│ │ ...                                      │ │
│ └──────────────────────────────────────────┘ │
│                                              │
│ 版本历史                                     │
│                                              │
│ v4  DeepSeek · 2h 前 · 并行修改              │
│ v3  你 · 3h 前 · 手动改了标题                │  ← 点版本 → Overlay 预览
│ v2  Claude · 5h 前 · review 加验证逻辑       │     点"应用此版本" → 当前变成它
│ v1  DeepSeek · 6h 前 · 首次生成              │
│     └─ 来源: "写个登录页面" → [跳到那条消息]  │
│                                              │
│ [重命名] [导出] [复制到别的房间] [删除]        │
└──────────────────────────────────────────────┘
```

### 版本规则

| 场景 | 行为 |
|------|------|
| AI 写了一个新版本 | 旧版本 → 快照保留。新版本 → 当前文件。 |
| 你手动改了文件 | 同上 |
| 两个 AI 同时改同一个文件 | 后完成的自动分支 (v4a / v4b)，文件列表显示"有并行修改待处理" |
| 你正在编辑 | 文件加锁。AI 生成完 → 存为"待审核"，不覆盖你正在改的 |
| 你选了一个旧版本点"应用" | 当前文件被旧版本覆盖。旧版本快照保留。 |

### 关联到对话

每个版本记录它出自哪条消息。v1 旁边有一行"来源: '写个登录页面' → [跳到那条消息]"。反过来，聊天里 AI 写文件时会出现一张文件卡片，点卡片直接进文件详情。**消息和文件是双向链接。** 谁生成了什么、谁改了什么，两边都能看到。

### 操作

| 操作 | 入口 | 权限 |
|------|------|------|
| 新建 | ＋ → 空白文件 / 从模板 | 任何人 |
| AI 写入 | 聊天里 AI 生成 → 人点"存" | 任何人 |
| 重命名 | 长按 → 重命名 | 任何人 |
| 导出 | 长按 → 导出 | 任何人 |
| 复制到别的房间 | 长按 → 复制到房间 → 选目标 | 任何人 (只复制当前版本) |
| 删除 | 长按 → 删除 → Sheet 确认 | 房间创建者 |

### 搜索

搜"登录" → 匹配文件名 Login.tsx、标签 #登录、版本描述"写个登录页面"。第一版不搜文件内容（太慢），以后加倒排索引。

## 三、资料 — 甲方发的、同事传的参考素材

### 用户场景

甲方发了三份 PDF 需求文档。张三传了两张竞品截图。你自己传了一份 Excel 数据表。这些都是项目的参考素材。放在这里，人和 AI 随时查阅。

一周后你想找"甲方发的那份关于支付流程的 PDF"。不记得文件名，只记得是甲方发的、关于支付的。

**你不是在翻文件夹——你是在翻记忆。**

### 默认视图：网格

```
┌─ 资料 ─────────────────────────────────────┐
│ [网格] [列表]    #全部 #甲方 #竞品 #数据     │
│                                      [上传] │
│                                            │
│ ┌──────────┐ ┌──────────┐ ┌──────────┐     │
│ │  🖼      │ │  📄      │ │  🖼      │     │
│ │ 首页参考  │ │ 支付流程  │ │ 详情页    │     │
│ │ 张三 · 2h │ │ 甲方 · 昨 │ │ 张三 · 3h │     │
│ └──────────┘ └──────────┘ └──────────┘     │
│                                            │
│ ┌──────────┐ ┌──────────┐ ┌──────────┐     │
│ │  📄      │ │  📄      │ │  📊      │     │
│ │ 需求v2   │ │ 接口说明  │ │ 数据表    │     │
│ │ 甲方·3天 │ │ 甲方·3天  │ │ 你 · 上周 │     │
│ └──────────┘ └──────────┘ └──────────┘     │
│                                            │
│ 6 个资料 · 12MB · 3 个已被 AI 分析          │
└────────────────────────────────────────────┘
```

### 点进一个资料

图片 → 全屏预览。底部显示上传者、时间、标签。AI 分析摘要可展开。可以编辑标签、导出、删除。

PDF → 内置 PDF.js 阅读器。翻页、搜索文字。侧边显示 AI 摘要。

### AI 自动分析

资料上传后，AI 自动读取并生成一段摘要。对于图片："这是一个电商 APP 首页设计稿。布局从上到下: 搜索栏、轮播 Banner、分类图标、商品推荐网格。" 对于 PDF："这是一份产品需求文档。主要内容包括: 用户注册登录、商品浏览搜索、购物车、支付流程。支付部分要求支持微信和支付宝。"

摘要两个用途：卡片上预览（不用打开就知道内容）、搜索时匹配（搜"支付" → 找到包含支付内容的 PDF）。

### 标签

上传时可选填标签。AI 分析完后自动建议 3 个标签，用户确认或修改。标签用于筛选（点顶部 #甲方 → 只看甲方发的）和搜索。

### 在聊天里引用

用户在输入框旁边点 + → 选"引用资料" → 弹出资料选择器 → 选一个 → 消息里嵌入资料卡片。AI 也可以在回复中引用资料："基于你上传的竞品截图，他们的首页..."

### 操作

| 操作 | 权限 |
|------|------|
| 上传 | 任何人 |
| 预览、导出、加标签 | 任何人 |
| 删除 | **上传者** + 房间创建者 |
| AI 改 | **不允许** |

### 和产出的关键区别

产出是人和 AI 一起**编辑**的东西。资料是只读的参考物。AI 能读资料、能分析资料、能在聊天里引用资料——但不能改资料本身。

## 四、归档 — Cron 每天自动生成的报告

### 用户场景

设了两个 Cron："每天早上 8:30 汇总未读邮件"、"每周一检查腾讯云账单"。它们自动运行，产出每天/每周的报告。

三周后，他想回头看 7 月 15 号的邮件汇总——那天有一封重要邮件需要追溯。他不记得那天的文件名，只记得"大概是 7 月中旬"。

**他不是在浏览文件——他是在翻日历。**

### 视图：按来源分组 + 日期

```
┌─ 归档 ───────────────────────────────────┐
│                                          │
│ ┌─ 每日邮件摘要 ───────────────────────┐ │
│ │ 来源: Cron · 每天 8:30               │ │
│ │ 保留: 最近 30 天 · 当前 22 份         │ │
│ │ 状态: ✅ 正常 · 今天已产出             │ │
│ │                                      │ │
│ │   ← 7月      今天 ↓                  │ │
│ │                                      │ │
│ │  7月21日   📊 3封未读 · 8:32         │ │
│ │  7月20日   📊 5封未读 · 8:30         │ │
│ │  7月19日   📊 2封未读 · 8:31         │ │
│ │  … 还有 19 份                        │ │
│ └──────────────────────────────────────┘ │
│                                          │
│ ┌─ 腾讯云账单 ──────────────────────────┐ │
│ │ 来源: Cron · 每周一 10:00             │ │
│ │ 状态: ⚠️ 上次失败 · 7月14日           │ │
│ │                                      │ │
│ │  7月14日   ⚠️ 执行失败               │ │
│ │  7月7日    📊 消费¥1,234 · 2.1KB     │ │
│ └──────────────────────────────────────┘ │
│                                          │
│ 2 个来源 · 26 份归档                      │
└──────────────────────────────────────────┘
```

### 失败处理

Cron 失败的那天，列表里显示 ⚠️ 执行失败 + 原因 + [重试] 按钮。同一个 Cron 连续失败 3 次 → 运行 tab 和房间顶栏出现醒目提醒。

### 保留策略

用户在 Cron 任务设置里配："保留最近 30 天"。每天 Cron 运行前检查 → 超期的自动删。用户也可以手动清空一个来源、或删除单份归档。

### 核心原则

归档是只读的。它是报告，不该被编辑。编辑没意义——下次运行就覆盖了。不需要标题（标题 = 来源名 + 日期）。不需要组织（按来源自动分组）。不需要版本（每次新文件）。需要过期——三个月的每日汇总没人看。

## 五、模板 — 做过一次、下次还用的格式

### 用户场景

产品经理做了三个项目房间，每个都要写 PRD。他在房间 A 里写了一份"PRD 模板"。两周后开房间 B——又得写 PRD。他需要一个**跨房间的模板库**。"我做过一次的东西，下次用的时候能找到。"

### 视图

```
┌─ 模板库 ───────────────────────────────────┐
│ [全部] [产品] [技术] [管理]    🔍 搜索…    │
│                                            │
│ ┌─── PRD 模板 ───────────────────────────┐ │
│ │ 📄 · v2 · 你 · 2 周前                  │ │
│ │ 产品需求文档标准模板, 含功能列表、       │ │
│ │ 优先级矩阵、评审清单                    │ │
│ │ #产品 #文档 · 使用 5 次                 │ │
│ │ [使用] [预览] [编辑]                   │ │
│ └────────────────────────────────────────┘ │
│                                            │
│ ┌─── 周报模板 ───────────────────────────┐ │
│ │ 📄 · v1 · 使用 3 次                    │ │
│ │ [使用] [预览] [编辑]                   │ │
│ └────────────────────────────────────────┘ │
│                                            │
│ [新建模板] [从现有文件保存为模板]            │
└────────────────────────────────────────────┘
```

### 使用模板

点 [使用] → 选目标房间 → 自定义文件名 → 创建。模板内容复制到房间的"产出"列表里。模板的 usageCount +1。

### 编辑模板

创建者可以编辑模板。编辑后版本 +1。**已使用此模板创建的文件不受影响**——种子改了，已经长出来的树不变。

### 也可以从现有文件创建

在产出 tab 长按一个文件 → "保存为模板" → 文件名自动填入 → 确认。

### 跨房间

模板不属于任何房间。所有房间都能使用。这是它和产出/资料/归档最根本的区别。

## 六、个人 — 自己的笔记、歌、健身记录

### 四种个人内容

| | 笔记 | 音乐 | 视频 | 健身 |
|------|------|------|------|------|
| **是什么** | 想法、灵感、日记 | MP3/FLAC 或流媒体 | 视频文件或 URL | 每天的训练记录 |
| **怎么组织** | 时间倒序，可固定 | 歌单 | 列表 + "稍后观看" | 月历 |
| **有多大** | <50KB/条 | 几MB到几十MB | 几十MB到几百MB | <200B/条 |
| **需要什么** | 搜索、固定 | 播放器、歌单 | 播放器 | 日历、周汇总 |

### 笔记

和备忘录 App 一样。列表按时间倒序，支持固定和搜索。点 + 进入全屏 Markdown 编辑器，停写 2 秒自动保存。长按可以"发送到房间"——笔记内容变成一条聊天消息，在房间里继续讨论。笔记是私有的，但用户可以主动把它升级为协作。

### 音乐

音乐文件存在 `/sdcard/music/`。歌单只存文件路径的 JSON 列表——不复制音频。用户可以自己建歌单、往里面加歌。

### 视频

简单的列表，按添加时间倒序。点开全屏播放。支持"稍后观看"标记。

### 健身

月历视图。每天有运动就显示图标（🏃 跑步、💪 力量）。点一天看详情。点 + 记录今天练了什么。底部显示本周和本月汇总。

### 个人和协作的边界

个人内容是私有的。笔记可以"发送到房间"变成聊天消息——但这个操作是用户主动触发的，不是自动的。**没有用户的明确动作，个人和协作之间没有数据流动。**

删 MOV → `/sdcard/mov/personal/` 保留。删看板应用 → 数据保留。

## 八、技术实现摘要

### 磁盘布局

```
/sdcard/mov/
  .mov/device.json                      ← 设备配置
  users/index.json                      ← 本地用户
  templates/                            ← 模板 (跨房间)
  board-apps/<app-id>/data/             ← 看板应用数据
  personal/                             ← 个人内容
    notes/  music/  videos/  fitness/
  rooms/<id>/
    chat/<date>.json                    ← 聊天 (每天一个文件, 只追加)
    chat/index.json                     ← 聊天索引
    files/work/                         ← 产出 (当前版本)
    files/work-snapshots/               ← 产出历史版本
    files/inbox/                        ← 资料
    files/archive/                      ← 归档 (按来源分目录)
    files/.meta/index.json              ← 一份元数据管所有文件
    .mov/config.json                    ← 房间配置
```

### 读写性能目标

| 操作 | 目标 | 方式 |
|------|------|------|
| 打开 MOV → 首屏 | <100ms | localStorage 同步读房间列表 |
| 进房间 | <50ms | 2 次文件读: index.json (~3KB) + 今日 chat (~40KB) |
| 打开文件 tab | 0ms | index.json 已在内存 |
| 发一条消息 | <50ms | 追加一行到 chat 文件 |
| 搜索 (跨房间) | <500ms | 暴力 grep + 以后加倒排索引 |

### 关键设计决策

1. 聊天消息从 localStorage 搬到文件系统。localStorage 只保留 <2KB 的房间列表摘要。
2. 文件元数据从每个文件一个 JSON 合并为一房间一份 index.json。进房间一次读。
3. 聊天按天切片——追加写不读全量。
4. SharedPreferences 保留 AI/Cron/技能配置（高频写、小数据）。
5. 五种存储类型在 JS 层各有独立的渲染和操作逻辑，底层复用通用读写 API。

## 1. 产品模型

```
一个房间 (Room)
  = 一个项目仓库
  = 讨论 (Discussion) + 文件 (Files)
  = 人类成员 + AI 成员 + MOV 主持

磁盘:
```

## 3. 新增操作

### 3.1 讨论中引用文件

当 AI 执行 `file.write` 时:
1. `CapabilityExecutor.doFileWrite(roomId, path, content)` 写入磁盘
2. `MOVBridge` 回调 JS: `_fileWritten(roomId, path, size, author)`
3. JS 在聊天区插入文件卡片消息 + 自动持久化到 msgData
4. 文件 tab 的树刷新

### 3.2 文件操作

| 操作 | 入口 | 行为 | 反馈 |
|------|------|------|------|
| 查看 | 点击文件名 | 文本: 弹窗或新 view 显示内容; 图片: 全屏查看 | — |
| 下载/导出 | 长按 → "导出" | 复制到 `/sdcard/Download/` | toast "已导出到下载目录" |
| 重命名 | 长按 → "重命名" | sheet 输入框 → 确认 | toast + 树刷新 |
| 删除 | 长按 → "删除" | sheet 确认 → `file.delete` | toast "已删除" |
| 新建文件 | 右上角 ＋ | sheet: 输入文件名 + 内容 | toast + 树刷新 + 讨论区通知 |
| 上传 | 右上角 ＋ → "从设备选择" | `B.pickFile` → 复制到房间目录 | toast + 树刷新 |

### 3.3 新建房间时的文件初始化

## 7. 小结: 改前 vs 改后

```
改前                              改后
────────────────────────────────────────────
房间 = 聊天记录                    房间 = 文件仓库 + 讨论
AI 输出 = 纯文本                   AI 输出 = 文件 (代码/文档/图片)
"交付物" = 假卡片                  "交付物" = 磁盘里真实的文件
聊天和文件无关                     聊天里引用文件, 文件有提交记录
```

这个架构一旦建立, MOV 就从"带 AI 的聊天工具"变成了"AI 参与的项目仓库"。聊天、Council、Cron 产出的东西都有地方存放——在房间的文件库里。
